# Text2Humanoid 重构计划

本文描述 Text2Humanoid 后续的工程重构方向。

当前项目已经跑通：

```text
FloodDiffusion
  -> MakeTrackingEasy
  -> BFM-Zero tracking_online
  -> MuJoCo G1
```

下一阶段的目标不是重写系统，而是在保持 demo 可运行的前提下，把现有 vibe coding 代码逐步收敛成更清晰、更稳定、更适合 streaming chunk pipeline 的工程结构。

截至当前版本，流式主链路已经有 research prototype：

```text
FloodDiffusionStreamingBackend
  -> StreamingRetargetBridge
  -> StreamingBFMZeroPublisher / BFMZeroFrameBuffer
  -> Demo Console Start Stream / Update Text / Stop Stream
```

因此后续重构重点不是重新打通功能，而是围绕交付质量做结构收敛、contract 命名统一、入口分级、测试与文档校准。

---

## 1. 重构目标

本次重构的核心目标是：

```text
让已经跑通的 demo 变成可维护、可扩展、可诊断的 streaming research demo。
```

具体包括：

* 清理仓库视野，移除运行产物和缓存噪音；
* 明确正式入口、开发工具、诊断工具的边界；
* 将 README 中的 streaming buffer 设计落实到代码结构中；
* 用 `chunk_duration_sec` 作为 streaming pipeline 的核心时间单位；
* 明确 generation / retarget / runtime 三段 buffer 边界；
* 统一 artifact metadata；
* 加强 chunk / FPS / joint order / quaternion order 的 contract test；
* 为后续 FloodNet backend、文本切换和 continuous streaming 留出干净接口。

本次重构不追求：

* 推倒重写所有代码；
* 立即实现完整低延迟 streaming；
* 立即解决所有 retarget 质量问题；
* 立即支持 sim2real；
* 立即删除所有历史代码。

---

## 2. 当前主要问题

### 2.1 仓库视野噪音较多

当前 tree 中混入了大量运行产物：

```text
__pycache__/
artifacts/
assets/saved/
assets/video/
*.mp4
*.npz
*.npy
*.log
*.keys
```

这些文件让项目看起来比实际复杂很多，也会干扰新成员理解源码结构。

目标：

* 源码、配置、工具、测试和运行产物分离；
* 默认 `tree` 能清楚展示工程结构；
* 大型或临时 artifact 不进入 git 视野。

---

### 2.2 apps 入口过多

当前 `apps/` 同时包含：

* 正式 demo 入口；
* replay 工具；
* retarget 转换工具；
* debug bundle 导出；
* 诊断工具；
* session / API 实验入口。

这会导致新成员不知道应该从哪个文件启动。

目标：

```text
apps/ 只保留稳定入口
tools/ 放诊断、转换、replay 工具
scripts/ 放一次性开发脚本
```

---

### 2.3 streaming 主线和离线验证路径混在一起

当前代码中存在两类路径：

```text
离线验证路径:
  generate full motion -> save artifact -> retarget -> replay

streaming 目标路径:
  generate chunk -> retarget chunk -> append runtime future buffer
```

离线 artifact 对调试很重要，但不应该成为架构主线。

目标：

* README 和代码命名中明确区分 `debug/offline validation` 与 `streaming pipeline`；
* 保留离线 artifact，但将其视为 streaming pipeline 的 debug checkpoint；
* 主接口围绕 chunk 和 buffer 设计。

---

### 2.4 FPS / chunk size 语义需要统一

当前上游和下游 FPS 不一致：

```text
FloodDiffusion / generation: 20 FPS
MakeTrackingEasy retarget:   30 FPS
BFM-Zero tracking_online:    50 FPS
```

后续不应使用模糊的 `chunk_size = 20` 表达 streaming 单元，而应使用：

```text
chunk_duration_sec
generation_fps
retarget_fps
runtime_fps
```

目标：

```text
chunk_duration_sec = 1.0
-> 20 frames @20 FPS
-> 30 frames @30 FPS
-> 50 frames @50 FPS
```

系统真正对齐的是时间，不是原始帧数。

---

### 2.5 buffer 边界尚未完全工程化

README 中已经明确 streaming buffer 链路：

```text
Generation Request Queue
  -> 263D Motion Buffer
  -> Retarget Queue
  -> G1 Motion Buffer
  -> Runtime Future Buffer
  -> BFM-Zero ZMQ Sender
```

后续需要逐步在代码结构中把这些边界显式化。

目标：

* generation 只产出 263D chunk；
* retarget 只消费 263D chunk 并产出 G1 chunk；
* runtime 只消费 G1 / BFMZero chunk 并维护 50Hz future buffer；
* orchestrator 负责 buffer watermarks、chunk request、文本切换和错误处理。

---

## 3. 重构原则

### 3.1 不破坏已跑通 demo

任何重构都必须保证以下入口仍可运行：

```text
apps/demo_console_server.py
apps/app_stop.py
```

如仍保留 CLI 验证入口，也应保证：

```text
apps/run_text_to_bfmzero.py
apps/launch_full_demo.py
```

每次移动文件或修改接口后，都需要运行对应 smoke test。

---

### 3.2 先降噪，再重构

优先顺序：

```text
1. 清理 __pycache__、artifacts、assets 输出
2. 整理 .gitignore
3. 标记稳定入口和开发工具
4. 再移动代码
5. 最后拆分内部包结构
```

不要一开始就大规模改 import。

---

### 3.3 以时间长度定义 chunk

所有 streaming 配置优先使用：

```text
chunk_duration_sec
```

不要只使用：

```text
chunk_size
```

如果代码中必须保留 frame 数，应明确命名：

```text
generation_chunk_frames
retarget_chunk_frames
runtime_chunk_frames
```

---

### 3.4 contracts 优先稳定

所有跨模块数据必须通过 contract 表达，避免直接传裸 dict 或裸 np.ndarray。

核心 contract：

```text
GeneratedMotionChunk
RetargetInputChunk
RobotMotionChunk
BFMZeroMotionChunk
TrackerStatus
```

每个 contract 应显式记录：

```text
representation
fps
shape
duration
chunk_id
start_time
source_chunk_id
joint_order
quat_order
metadata
```

---

### 3.5 runtime 不修复动作语义

runtime 层只负责：

* 50 FPS 重采样；
* joint order 转换；
* quaternion order 转换；
* velocity 计算；
* frame_idx 连续性；
* ZMQ 发送；
* future buffer 管理。

runtime 不负责：

* 修 root orientation；
* 修脚接触；
* 修 retarget 跳变；
* 修 generation 语义错误。

这些问题应在 retarget / diagnostics 层处理。

## 5. 分阶段重构计划

## Phase 0: 仓库降噪

目标：先让项目看起来干净。

任务：

* 删除所有 `__pycache__/`；
* 删除或移出临时 `artifacts/`；
* 删除或移出 `assets/saved/`、`assets/video/`；
* 确认大型 `.mp4`、`.npz`、`.npy`、`.pt`、`.log` 不进入 git；
* 增加 / 更新 `.gitignore`；
* 保留最小 fixture，例如 `tests/fixtures/` 或 `assets/fixtures/`。

建议 `.gitignore`：

```gitignore
# Python
__pycache__/
*.py[cod]
.pytest_cache/
.mypy_cache/
.ruff_cache/

# Runtime artifacts
artifacts/
assets/saved/
assets/video/
assets/save/
*.log
*.keys

# Heavy local outputs/checkpoints
*.mp4
*.pt
*.pth

# Keep small fixtures versionable
!tests/fixtures/**/*.npy
!tests/fixtures/**/*.npz
!assets/fixtures/**/*.npy
!assets/fixtures/**/*.npz
```

验收：

* `tree -L 3` 能清楚看到源码结构；
* git status 不显示运行产物；
* demo 仍能生成新的 artifact。

---

## Phase 1: 稳定入口分级

目标：明确哪些是用户入口，哪些是开发工具。

保留在 `apps/`：

```text
demo_console_server.py
app_stop.py
launch_full_demo.py
run_text_to_bfmzero.py
api_server.py    # 如果仍计划维护 API
```

建议移动到 `tools/diagnostics/`：

```text
diagnose_263_to_mte.py
export_debug_bundle.py
```

建议移动到 `tools/replay/`：

```text
replay_bfmzero_chunk.py
replay_bfmzero_reference.py
replay_reference.py
replay_trajectory.py
```

建议移动到 `tools/conversion/`：

```text
retarget_motion.py
retarget_to_bfmzero.py
```

验收：

* README 只列稳定入口；
* tools 中的脚本仍可通过 `PYTHONPATH=src` 运行；
* tests 中对应 CLI 路径同步更新。

---

## Phase 2: 明确 streaming 配置语义

目标：把 chunk 的主语义从 frame count 改为 duration。

任务：

* 在 config 中加入：

```yaml
streaming:
  chunk_duration_sec: 1.0
  generation_fps: 20
  retarget_fps: 30
  runtime_fps: 50
  low_watermark_sec: 1.0
  target_buffer_sec: 2.0
```

* 避免使用模糊的 `chunk_size`；
* 如必须使用 frame 数，使用派生字段：

```text
generation_chunk_frames
retarget_chunk_frames
runtime_chunk_frames
```

* 所有 artifact metadata 记录：

```text
chunk_duration_sec
source_fps
target_fps
frame_count
duration_sec
```

验收：

* 给定 `chunk_duration_sec = 1.0`，能明确得到：

  * 20 generation frames；
  * 30 retarget frames；
  * 50 runtime frames；
* tests 覆盖 FPS / duration 换算；
* README 与 config 命名一致。

兼容策略：

* 第一版新增 `streaming` 配置，但保留旧的 `planner.chunk_frames`、`runtime.*_frames` 字段；
* 现有 `configs/system/*.yaml` 的 `streaming` 值按旧字段等价换算，避免改变当前 demo 行为；
* 如果某个旧配置没有 `streaming` 段，代码仍从旧字段派生 timing 语义。

---

## Phase 3: contracts 收敛

目标：所有跨模块数据通过明确 contract 传递。

任务：

* 梳理现有 `contracts/`；
* 明确是否保留旧 `GeneratedMotion`，还是统一命名为 `GeneratedMotionChunk`；
* 为以下对象增加 / 检查字段：

```text
chunk_id
source_chunk_id
start_time
duration_sec
fps
representation
motion_shape
joint_order
quat_order
metadata
```

* 为 shape / fps / quaternion order 增加校验函数；
* 减少跨模块裸 dict 和裸 np.ndarray。

验收：

* generation 输出必须是 `GeneratedMotionChunk`；
* retarget 输入必须是 `RetargetInputChunk`；
* runtime 输入必须是 `RobotMotionChunk` 或 `BFMZeroMotionChunk`；
* tests 覆盖 contract shape check。

---

## Phase 4: generation buffer 边界

目标：让 generation 层成为明确的 263D chunk producer。

任务：

* 明确接口：

```python
generate_chunk(request) -> GeneratedMotionChunk
stream_chunks(request) -> Iterator[GeneratedMotionChunk]
```

* `generate_chunk()` 保留为 debug / validation path；
* `stream_chunks()` 作为 streaming 主接口；
* 统一 chunk_id / start_time / duration_sec；
* FloodDiffusion subprocess 输出 artifact 时，仍包装成 `GeneratedMotionChunk`；
* 避免 generation 层知道 BFM-Zero / G1 / ZMQ。

验收：

* 不启动 retarget，也能生成合法 `GeneratedMotionChunk`；
* generation artifact metadata 完整；
* 单测覆盖 prompt -> chunk contract；
* streaming fake backend 可连续产出多个 chunk。

---

## Phase 5: retarget buffer 边界

目标：让 retarget 层成为明确的 263D chunk consumer 和 G1 chunk producer。

任务：

* 明确接口：

```python
convert_chunk(GeneratedMotionChunk) -> RetargetInputChunk
retarget_chunk(RetargetInputChunk) -> RobotMotionChunk
```

* 20 FPS -> 30 FPS 重采样按 duration 对齐；
* 保留必要的 previous chunk tail；
* 明确 root position / root yaw continuity check；
* MakeTrackingEasy 输出统一包装为 `RobotMotionChunk`；
* retarget diagnostics 不混入 runtime sink。

验收：

* 给定 1s 263D chunk，输出 1s G1 chunk；
* 20 -> 30 FPS 帧数正确；
* joint order / quat order 被写入 metadata；
* chunk 边界 continuity check 有测试或诊断报告；
* retarget 异常不会导致 runtime 发送脏数据。

---

## Phase 6: runtime future buffer

目标：让 runtime 成为 50Hz fixed-rate consumer。

任务：

* 明确 `BFMZeroMotionChunk`；
* 明确 `RuntimeFutureBuffer` 或等价类；
* append chunk 时生成连续 frame_idx；
* 30 FPS -> 50 FPS 重采样按 duration 对齐；
* 在 50 FPS 上重新计算：

  * joint_vel；
  * root_lin_vel_w；
  * root_ang_vel_w；
* 明确 low_watermark / target_buffer；
* buffer underrun 时进入明确 fallback / stop 状态；
* ZMQ sink 只消费 future buffer，不直接做 retarget 修复。

验收：

* 给定 1s RobotMotionChunk，输出 50 frames；
* frame_idx 连续；
* joint_vel / root velocities shape 正确；
* mock subscriber 能接收并解析 frame；
* buffer low watermark 能触发 next chunk request；
* buffer underrun 有明确状态。

---

## Phase 7: orchestrator streaming 调度

目标：让 orchestrator 管理 session-level streaming，而不是只跑单段 pipeline。

任务：

* 明确 session state：

```text
current_prompt
pending_prompt
next_chunk_id
playhead_frame
future_buffer_level
generation_inflight
retarget_inflight
runtime_state
```

* 实现基本调度策略：

```text
if future_buffer < low_watermark:
    request next chunk
```

* 支持 first chunk bootstrap；
* 支持 stop / reset；
* 支持 generation / retarget error propagation；
* 文本切换先支持 chunk boundary 切换；
* 不承诺无缝 crossfade。

验收：

* demo console 可以启动一个 session；
* session 至少能连续 append 多个 chunk；
* buffer low watermark 触发后续 chunk；
* pending prompt 能在 chunk boundary 生效；
* session stop 后进程和 buffer 清理干净。

## 6. 推荐优先级

推荐执行顺序：

```text
P0:
  Phase 0 仓库降噪
  Phase 1 入口分级
  Phase 2 streaming 配置语义

P1:
  Phase 3 contracts 收敛
  Phase 6 runtime future buffer
  Phase 8 测试和文档

P2:
  Phase 4 generation buffer
  Phase 5 retarget buffer
  Phase 7 orchestrator streaming 调度

P3:
  文本切换
  overlap / crossfade
  FloodNet backend
  远程 generation backend
```

为什么 runtime future buffer 优先级较高：

* BFM-Zero 50Hz 是下游硬约束；
* 只要 runtime 输入契约稳定，上游 generation / retarget 可以逐步替换；
* mock chunk 可以先绕过重模型测试 runtime；
* 这能最快降低系统不确定性。

## 7. 风险点

### 7.1 大规模移动文件导致 import 破坏

缓解：

* 先移动 tools，不动核心包；
* 每次移动后跑测试；
* 保留兼容 wrapper 一段时间。

---

### 7.2 streaming 设计写得很理想，但代码仍是离线单段

缓解：

* README 明确区分“当前实现”和“目标设计”；
* refactor plan 中分 phase 落地；
* 离线路径保留为 validation path。

---

### 7.3 retarget 边界质量问题被 runtime 掩盖

缓解：

* runtime 不做语义修复；
* retarget 输出前做 diagnostics；
* root orientation / root angular velocity 单独报告。

---

### 7.4 FPS 对齐错误

缓解：

* 所有阶段使用 `duration_sec` 对齐；
* 添加 FPS/duration 测试；
* artifact metadata 记录 source_fps / target_fps。

---

## 8. 完成后的理想状态

完成第一轮重构后，项目应该具备以下特征：

```text
README 能解释系统；
docs/refactor_plan.md 能指导重构；
apps/ 只有稳定入口；
tools/ 是清晰的诊断 / replay / conversion 工具；
contracts 定义跨模块数据；
generation 只产出 263D chunk；
retarget 只产出 G1 chunk；
runtime 只维护 50Hz future buffer 和 ZMQ；
orchestrator 负责 session streaming 调度；
artifact metadata 能复现实验；
tests 能防止 FPS / joint order / quat order / frame_idx 回归。
```

最终目标不是让代码“看起来很工程化”，而是让下一位研究伙伴能够安全地修改 generation、retarget 或 runtime，而不破坏完整 demo。
